## What Went Well

**Architecture-first design paid off.**
The pre-implementation architecture phase resolved structural decisions before code was written. No story had to reverse a structural choice mid-implementation. The AJV bundling strategy (devDependencies for tsup to bundle into dist), the grammar type disambiguation (`StoryModel.grammar: string[]` vs `SentenceModel.grammar: number[]`), and the Turborepo pipeline dependency model all executed cleanly because they were resolved in planning.

**The review process caught real bugs.**
The code review was effective on every story. Notable catches:
- Story 1.3: `version in LOADERS` prototype chain vulnerability (HIGH) ŌĆö `"constructor"` as a schema version would have bypassed the guard
- Story 1.5: 19 CSV entries with commas in meaning fields silently assigned to "Genki I Intro" instead of their correct chapters (Genki I Ch.10, Ch.12, Genki II Ch.14, etc.) ŌĆö caught immediately when the review patch added `console.warn` on unexpected column counts

**Zero structural regressions across stories.**
Each story built cleanly on the previous. The workspace symlinks, tsup build outputs, and type exports all composed correctly without backtracking.

## Challenges and Learnings

**pnpm install friction recurred in 3 of 5 stories.**
The esbuild postinstall script approval process (pnpm 11 security policy) caused install failures or unexpected exit codes in Stories 1.1, 1.2, and 1.4. The fix (`pnpm.onlyBuiltDependencies: ["esbuild"]` + `pnpm rebuild esbuild`) was documented but pnpm v11 still required manual intervention on new version installs. Not a blocker, but a recurring DX friction point.

**ESM/CJS boundary created friction at the infrastructure layer.**
Three stories touched this boundary: AJV (CommonJS library, needed careful devDependency placement), `vite.config.ts` (no `__dirname` in ESM context), and `postcss.config.js` (required `type: module`). Epic 2 is pure in-app React code with no new package boundaries ŌĆö this friction is expected to disappear.

**ESLint deferral became a quiet blocker.**
Deferring full ESLint config from Story 1.1 ("Story 1.5 scope") was the right call, but by Story 1.4 the absence of `apps/web/.eslintrc.cjs` meant `turbo lint` would have failed silently. The deferred item wasn't tagged clearly enough as "blocks Story 1.4 lint". Going forward, deferred items should explicitly name the story they block.

## Process Agreements

- When deferring a task, note explicitly which future story it blocks (e.g. "deferred ŌĆö blocks Story 1.4 lint")
- Code review patches are healthy signal, not waste ŌĆö no target to reduce them; quality over speed
- Build scripts that process external data should include explicit warnings on unexpected input shapes (validated by the build-vocab experience)
