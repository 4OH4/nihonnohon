## What Went Well

**Component architecture composed cleanly.**
Seven new components across five stories — WordToken, SentenceBlock, InfoPanel, KanjiBreakdown, AppBar, ToolBar, ReaderRoute — stacked without any structural rework. No story had to reverse a design decision made by a previous story.

**Test isolation pattern held across the entire epic.**
The `_initFromData`/`_reset` helper pattern established in Story 2.1 was reused consistently in every subsequent story's tests. Zero test pollution incidents.

**Review process caught real production bugs.**
- Story 2.1: Concurrent `initVocab`/`initKanji` callers both passing the null guard before either fetch resolves — last-write-wins overwrite. Would have been a subtle production bug under concurrent navigation.
- Story 2.3: Tailwind v3 `transition-property` conflict — when two `transition-*` classes are present, only the last applies. Silent visual regression caught before it shipped.
- Story 2.5: `aria-pressed` missing on toggle buttons — screen readers couldn't announce toggle state.

**Zero structural regressions.**
Each story built on the previous without reverting or restructuring shared code. Backward-compatible prop additions (supplement support in WordToken/SentenceBlock) worked cleanly.

## Challenges and Learnings

**Discoverability gap created a false impression of incompleteness.**
The LibraryRoute stub had no link to the reader, so navigating to the app showed "Story library — implemented in Epic 3." RT had to be told the direct URL to verify Milestone 1. The reader was fully functional — it just wasn't reachable from the UI.

*Action:* Every epic must leave the app navigable to what was just built, even via a hardcoded link in a stub. A demo path is part of the milestone definition.

*Immediate fix:* Added a `→ Open demo story` link to LibraryRoute stub and committed before closing the retro (`fd69c00`).

**`createMemoryRouter` + jsdom has async timing issues in React Router v6 tests.**
The recommended `createMemoryRouter` + `RouterProvider` pattern produced empty renders in Story 2.5 — route content never appeared after the loader resolved. Root cause: jsdom's async execution model doesn't flush React Router's internal navigation promise chain reliably. Workaround: `vi.mock('react-router-dom')` to mock `useLoaderData`, wrapped in `MemoryRouter` for `Link` context. This is now the established pattern for route component tests.

*Action:* Document in `project-context.md` before Story 3.1.

**Epic 1 retro field names drifted from final implementation.**
The Epic 1 retro documented the planned `KanjiEntry` shape as `character/meaning/onYomi/kunYomi`. The actual implemented type was `char/kw/m/onY/kunY`. Story 2.1 dev notes had to correct for this. Retro documents capture planning intent, not final implementation state — treat them as point-in-time snapshots.

## Process Agreements

- **Demo path rule:** Every epic closes with the app navigable to its deliverables from the UI, even via a temporary hardcoded link. This is part of the milestone definition.
- **React Router v6 test pattern:** Route component tests use `vi.mock('react-router-dom')` + `MemoryRouter`, not `createMemoryRouter`. Document in `project-context.md`.
- **Retro docs are point-in-time snapshots:** Do not treat them as living specifications. If types or field names change post-retro, the retro doc is not updated.
