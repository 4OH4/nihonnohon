# Story se2-3: Schema and Type Updates

Status: done

## Story

As a **developer**,
I want the `vocabEntry` schema and TypeScript types to include optional `pos` and
`dictionary_form` fields,
So that enriched vocabulary entries generated by the new pipeline can carry POS and
lemma information for future UI and tooling use.

## Acceptance Criteria

1. **AC1 — JSON Schema: `vocabEntry` gains optional `pos` and `dictionary_form`**
   - `packages/schema/schemas/story.v2.json` `$defs.vocabEntry.properties` gains:
     - `"pos": { "type": "string" }` (optional — NOT in `required`)
     - `"dictionary_form": { "type": "string" }` (optional — NOT in `required`)
   - Both fields appear in `properties`; `required` array is unchanged
     (`["key", "word", "hiragana", "translation"]`)
   - A story JSON with `vocab_supplement` entries that **omit** `pos` and `dictionary_form`
     passes AJV validation — fields are optional and absence is not an error

2. **AC2 — TypeScript: `VocabSupplementEntry` gains optional camelCase fields**
   - `packages/schema/src/types.ts` `VocabSupplementEntry` gains:
     - `pos?: string`
     - `dictionaryForm?: string`
   - No existing required fields changed

3. **AC3 — Loader: `v2.ts` maps new optional fields**
   - `packages/story-loader/src/v2.ts` `WireVocabEntry` gains optional
     `pos?: string` and `dictionary_form?: string`
   - `mapVocabEntry` maps `dictionary_form` → `dictionaryForm` when present;
     maps `pos` → `pos` when present; absence of either field is handled
     gracefully (no error, field absent on output object)

4. **AC4 — Changelog updated**
   - `packages/schema/SCHEMA_CHANGELOG.md` documents the additive change to
     `vocabEntry`: new optional `pos` and `dictionary_form` fields, their source
     (SudachiPy + enrichment pipeline), and that existing stories without these
     fields remain valid

5. **AC5 — Typecheck passes**
   - `turbo typecheck` exits 0 after all changes

## Dev Notes

### Critical dependency: se2-2 output requires this story

`EnrichmentPipeline.build_enriched_story` (se2-2) already emits `vocab_supplement` entries
with `pos` and `dictionary_form` fields. The JSON schema currently has
`"additionalProperties": false` on `vocabEntry`. Without this story, any story generated by
the new pipeline will **fail AJV validation** in `loadV2()` when the web app tries to load it.
This story unblocks production use of se2-2.

### Task 1: Exact JSON Schema change

Current `$defs.vocabEntry.properties`:
```json
{
  "key": { "type": "integer", "minimum": 0 },
  "word": { "type": "string" },
  "hiragana": { "type": "string" },
  "translation": { "type": "string" }
}
```

New (add two optional fields):
```json
{
  "key": { "type": "integer", "minimum": 0 },
  "word": { "type": "string" },
  "hiragana": { "type": "string" },
  "translation": { "type": "string" },
  "pos": { "type": "string" },
  "dictionary_form": { "type": "string" }
}
```

Do NOT touch `"required"` — it stays `["key", "word", "hiragana", "translation"]`.
`"additionalProperties": false` remains — the new fields are explicitly listed in
`properties` so they are permitted.

### Task 2: Exact TypeScript change

Current `VocabSupplementEntry`:
```typescript
export interface VocabSupplementEntry {
  key: number
  word: string
  hiragana: string
  translation: string
}
```

New:
```typescript
export interface VocabSupplementEntry {
  key: number
  word: string
  hiragana: string
  translation: string
  pos?: string
  dictionaryForm?: string
}
```

Note: `VocabSupplementEntry` is used for BOTH `keywords` and `vocabSupplement` on `StoryModel`.
The new optional fields apply to both, which is correct — `keywords` is legacy and won't have
these fields, but optional means absence is fine.

### Task 3: Exact v2.ts change

Current `WireVocabEntry`:
```typescript
interface WireVocabEntry {
  key: number
  word: string
  hiragana: string
  translation: string
}
```

New:
```typescript
interface WireVocabEntry {
  key: number
  word: string
  hiragana: string
  translation: string
  pos?: string
  dictionary_form?: string
}
```

Current `mapVocabEntry`:
```typescript
function mapVocabEntry(e: WireVocabEntry): VocabSupplementEntry {
  return { key: e.key!, word: e.word, hiragana: e.hiragana, translation: e.translation }
}
```

New (spread existing fields, conditionally add optionals):
```typescript
function mapVocabEntry(e: WireVocabEntry): VocabSupplementEntry {
  const entry: VocabSupplementEntry = {
    key: e.key!,
    word: e.word,
    hiragana: e.hiragana,
    translation: e.translation,
  }
  if (e.pos !== undefined) entry.pos = e.pos
  if (e.dictionary_form !== undefined) entry.dictionaryForm = e.dictionary_form
  return entry
}
```

Alternative (simpler with spread — also correct since TypeScript optional props allow undefined):
```typescript
function mapVocabEntry(e: WireVocabEntry): VocabSupplementEntry {
  return {
    key: e.key!,
    word: e.word,
    hiragana: e.hiragana,
    translation: e.translation,
    ...(e.pos !== undefined && { pos: e.pos }),
    ...(e.dictionary_form !== undefined && { dictionaryForm: e.dictionary_form }),
  }
}
```

Either form is acceptable. The important invariant: absence of the field on the wire object must
NOT produce `pos: undefined` on the output (TypeScript optional means the key should be absent,
not present with value `undefined`). Use conditional spread or explicit if-guard.

### Architecture rules that apply

From `project-context.md`:
- Wire format is **snake_case**; TypeScript model is **camelCase** — transformation happens
  exclusively inside the loader (`v2.ts`), nowhere else. `dictionary_form` in JSON →
  `dictionaryForm` in TypeScript. This is the exact pattern needed here.
- `schema_version` is an integer-as-string (`"1"`, `"2"`) — no version bump for additive
  optional field additions.
- TypeScript strict mode — no implicit any; `pos?: string` not `pos?: string | undefined`
  in the interface (TypeScript handles the difference).
