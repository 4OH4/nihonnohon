## What Went Well

**Always-mounted visibility pattern (`h-0 overflow-hidden`).** Established in Story 2.6 for `GenerationProgress`, propagated identically to `OutputPanel` (2.7) and the download toast (2.8). Zero layout shift across the generation cycle. The pattern was right first time and held across five stories without deviation.

**Injected `createEventSource` factory in `useAgUiRun`.** Architecture-specified in 2.1, honoured throughout. When Story 2.5 expanded the test suite to 16 tests, the `MockEventSource` injection point made every path testable. A testability seam that would have been painful to retrofit after the fact.

**Code review process found real defects every story.** 51 patches in 9 stories — not style feedback, actual logic errors. Highlights: Safari `revokeObjectURL` timing (2.8), `editedValue` not synced to store for download (2.7), debounce flush-on-unmount (2.9), `outputIsDirty` sanitization when `outputJson` is null (2.9). Each prevented a downstream production failure.

**State machine held up.** The 8-phase discriminated union from Story 2.1 (`idle`, `generating`, `cancelling`, `output-clean`, `output-dirty`, `downloading`, `error`, `proposal`) never required restructuring. The `downloading` phase handled the 2.8 save flow; the `proposal` phase placeholder is ready for Epic 4. Clean architectural foresight.

**`deferred-work.md` discipline.** ~51 items logged and committed to none prematurely. The registry of known-but-acceptable issues let the team move forward without pretending problems didn't exist.

**Story 2.5 as a dedicated test expansion story.** Treating test coverage of a complex SSE lifecycle hook as its own story (rather than bolting tests onto a feature story) produced 16 clean, comprehensive tests covering error paths, timeout paths, and cancel paths. The right model for a hook of that complexity.

**Test suite growth: 2 → 180 tests.** The test suite became a confidence multiplier that made every subsequent story faster and safer to implement.

## Challenges

**`act()` warning noise in tests.** Every test file generates React `act()` warnings from the Zustand `useAuthoringStore.subscribe()` / non-React side-effect interaction. Tests pass, but the console is consistently noisy. Training to ignore console warnings is a risk.

**`authoringStore.ts` grew larger than designed.** By Story 2.9 the store had 8 internal `_*` actions: `_setOutputJson`, `_setProposalText`, `_markDirty`, `_editOutputJson`, `_setError`, `_resolveCancel`, `_markRunStarted`, `_clearDownloadToast`, `_setSessionRestored`. The interface is coherent but larger than the original architecture specified. `_markDirty` and `_editOutputJson` now overlap in purpose.

**A2 (cross-platform `make dev`) carried forward unresolved.** The Windows/POSIX gap from Epic 1 was not addressed. With Epic 3's more complex backend (4-agent ReAct loop, more Makefile targets), this gap will become more painful. It's now a High priority item.

**Story 2.7's `_editOutputJson` was added post-review.** The initial implementation used `_markDirty()` alone, which didn't sync the edited value to the store for the Story 2.8 download. The review caught it, `_editOutputJson` was added — but it reveals that the Story 2.7 author didn't fully think through the 2.8 dependency. Cross-story dependency awareness could be sharpened.

## Key Insights

**1. The review process is the quality gate, not the ACs.** The acceptance criteria verified feature correctness. The multi-layer adversarial reviews (Blind Hunter, Edge Case Hunter, Acceptance Auditor) caught correctness bugs that ACs can't predict. ~51 patches across 9 stories is the evidence.

**2. Scope-bounding works at story level.** Story 2.8 deliberately excluded Genki vocab ID full validation (client-side can't verify without the CSV). The limitation was documented, the boundary was clean, and the backend already covered it. Not every validation needs to be complete on both sides.

**3. The session state design from the architecture doc was approximately right.** Story 2.9 implemented `useSession` largely as specified in the architecture doc. The main addition was `steeringInstructions` in the `hasContent` check — a gap the review caught. Architecture-level thinking about session state paid off.

**4. Plain JSON mode + post-hoc Pydantic validation > `response_schema` for Gemini.** The Epic 1 hypothesis was confirmed throughout Epic 2. The backend validates before `RUN_FINISHED` regardless; the structured output approach didn't add reliability value worth the RootModel/Enum limitations.

**5. `deferred-work.md` as a living document is more valuable than it looks.** Reviewers cite it, story authors consult it, and it prevents the same issue from being re-discovered. It's accumulated ~100+ items across both epics. Worth reviewing before each story context is written.
