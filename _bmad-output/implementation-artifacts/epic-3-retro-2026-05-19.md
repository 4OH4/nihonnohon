**`proposalApproved` error recovery design was clean.** When Japanese conversion fails, `_setError` checks `proposalApproved === true` and restores to `'proposal'` phase with the English draft intact. Correct first-try, clean state machine, no structural rework needed.

## Challenges

**Post-story Phase 2 routing bug.** After all stories were committed, RT discovered that clicking "Convert to Japanese" triggered phase 1 (English proposal generation) again instead of phase 2 (Japanese conversion). Root cause: `approve()` stores `topicText` in `storedInputs` alongside `englishDraft`; `useAgUiRun` sent both as URL params; the backend's `if path_mode == "B" and topic:` check fires before the `english_draft` check. Fix: `topic` and `englishDraft` are now mutually exclusive in the SSE URL.

**The bug was not caught by any review layer.** Blind Hunter, Edge Case Hunter, story AC tests — all passed. The unit tests verified each phase in isolation but not the interaction when `storedInputs` contained both fields simultaneously. A structural blind spot: AI code review layers only see diffs, not running system behaviour.

**Story 3.2 had the highest test friction.** 37 new tests, several requiring careful ordering: `setPathMode` must be called before `_setProposalText` because `setPathMode` has side effects that reset phase. This is an implicit ordering dependency that could catch future developers.

**`act()` warning noise — third consecutive carry.** Still unresolved, still noise. The test suite passes, but the console is consistently dirty. Two full epics of training to ignore warnings is a risk.

**Mutual exclusivity not explicit in contract documents.** The spec said "two separate SSE requests" but did not state "never send both `topic` and `englishDraft` simultaneously." The constraint was implied, not documented. Story 4.1 context must be explicit about what each phase sends and what it must omit.

---

## Key Insights

**1. Integration-level coverage is the gap AI code review can't fill.** Unit tests verify logic. Multi-layer adversarial review catches correctness bugs. But two-phase interaction bugs — where each phase is correct in isolation but wrong when parameters from both phases coexist — require a running system or an integration test that exercises the full cross-phase path.

**2. "Runnable and usable" is its own milestone.** Getting to a system you can actually operate reveals a class of issues that story ACs and code review cannot. Epic 3 crossed that threshold. The post-story bug was found by using the system, not by reviewing it.

**3. Explicit mutual exclusivity prevents ambiguous contracts.** Any story that introduces parallel parameter sets for different phases must state explicitly which params belong to which phase and cannot coexist. This is a story context authoring discipline, not a code discipline.

**4. The `proposalApproved` flag is a useful but fragile mechanism.** It correctly routes `_setError` to proposal recovery, but it also needs to be reset at `_setOutputJson` (caught by code review, Story 3.3 P1). Flag-based routing is effective for v1 but the coupling between `proposalApproved`, `_setError`, and `_setOutputJson` is non-obvious.

**5. Simple, targeted observability beats complex analytics for an authoring tool.** Elapsed time during generation is already shown. Preserving it after the run completes — displayed alongside the StatsBar — gives the author meaningful feedback without instrumenting a telemetry pipeline.
