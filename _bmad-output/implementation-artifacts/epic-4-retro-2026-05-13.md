## Challenges and Learnings

**CSS custom properties are a two-sided contract — both sides need tests.**
Story 4.3's highest-severity bug: `--story-font-size` was SET on the container, but the `fontSize: 'var(--story-font-size)'` consumer was accidentally removed during the ReaderRoute rewrite. Tests verified the store state (`textSize === 'large'`) but never asked whether the rendered element actually read the variable. The feature was completely non-functional visually — text size buttons produced no effect. Caught only in code review.

The fix was adding `element.style.getPropertyValue('--story-font-size')` to the test. The lesson: whenever a CSS custom property is introduced, test both the assignment AND the consumption.

**Firefox requires an explicit install — `playwright install` doesn't include it by default.**
AC6 required all 4 browser projects. The dev agent ran the suite without verifying Firefox was installed. The fix was `playwright install firefox` explicitly. This is a pre-flight verification gap, not a review gap.

**Playwright E2E has its own environment learning curve.**
Four separate issues surfaced during Story 4.4 implementation:
1. `__dirname` not available in `"type": "module"` ESM context → fix: `dirname(fileURLToPath(import.meta.url))`
2. `test.use()` cannot be called inside a nested `describe` block → fix: iPhone 14 coverage via the `mobile-safari` Playwright project
3. `axe-core` without tag scoping generates best-practice noise → fix: `withTags(['wcag2a', 'wcag2aa', 'wcag21a', 'wcag21aa'])`
4. Vocab/grammar tab tests timed out at default desktop width → fix: `setViewportSize({ width: 768 })` for mobile-specific tab bar

All were caught before merge. All four patterns are now known.

**Visual features need visual verification — not just store assertions.**
The text size bug, the missing `text-paper-text` token, the Space bar scroll behaviour — all of these would have been immediately obvious with the dev server running. Tests can verify state; only the browser verifies rendering. The dev-story workflow specifies starting the dev server for UI changes; this step was not consistently executed.

**Pattern: "caught in review" is not the same as "review went wrong."**
Review caught genuine bugs this epic — including the text size regression and the real WCAG violation. The goal isn't a deeper review pass; it's fewer defects reaching review by catching them earlier. The three targeted actions (CSS property patterns, browser pre-flight, visual verification) address the root causes directly.

## Process Agreements

- **CSS custom properties require two-sided tests.** When a CSS custom property is introduced, the test suite must assert both: (a) the property is assigned on the container element, and (b) the consuming property reads it. `element.style.getPropertyValue('--my-var')` is the assertion pattern.
- **Playwright browser pre-flight before final E2E task completion.** Before marking an E2E story done, verify all browser projects listed in `playwright.config.ts` are installed. `playwright install <browser>` is explicit; do not assume a prior install covered everything.
- **Visual verification is part of the green phase for UI stories.** For any story with CSS visual features, start the dev server and exercise the feature in a browser before marking the task complete. Store assertions alone are insufficient for rendered visual behaviour.
